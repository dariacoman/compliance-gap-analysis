Compliance Gap Analysis — Architecture Diagrams

Two architectures evaluated in this project. The simplified path is the production demo; the chain is the initial evaluated baseline that surfaced the failure modes documented in docs/evaluation-findings.md.

User-facing Shared infrastructure Deterministic step (no LLM) LLM call Silence detection (deterministic)

1 — Simplified architecture (production demo path)

Single LLM call per query. Implemented in src/simplified.py. Cost: ~3,000–5,000 tokens, ~30s per cold query on local CPU, ~5–10s on Colab GPU.

Input
User question
Compliance officer types a natural-language question, e.g.
"Have we performed a Fundamental Rights Impact Assessment under EU AI Act Article 27?"
Step 1 · Retrieve regulation chunks (no LLM)
BGE-large embedding + cosine top-k
Embed the query with BAAI/bge-large-en-v1.5 using BGE's recommended query-instruction prefix. Compute cosine similarity against pre-computed corpus embeddings (cached at embeddings_bge/). Filter corpus_tag == "REG"; return top-5 law chunks.
Step 2 · Retrieve deployer chunks (no LLM)
Same retriever, different filter
Same query embedding. Filter corpus_tag in ("DEP", "DEP_EXTRAS"); return top-5 deployer-side chunks. ICO operational guidance is excluded at this step — see evaluation-findings.md Stage 5/7 for the contamination rationale.
Step 3 · Single LLM call
Qwen 2.5-1.5B-Instruct (greedy decoding)
Construct prompt: SYSTEM_PROMPT (5 procedural rules) + user message containing the question, the 5 law chunks, the 5 deployer chunks, and a 3-section output template. Call once. do_sample=False, repetition_penalty=1.05, max_new_tokens=550. See decisions.md for decoding rationale.
Output
3-section gap analysis
Markdown text with three sections: "What the law requires", "What the policy says", "Gap". Each section cites chunk_ids verbatim in [square brackets]. Cached on (rendered_prompt, model_id) so re-runs of the same query are sub-second.

Total LLM calls: 1  ·  Token cost: ~3–5K  ·  Cold latency: ~30s on CPU  ·  Cached latency: < 1s

Why this shape: single LLM call avoids the multi-step chain's compounding error modes (decomposition drift, miscalibrated silence detection, mislabelled provenance). Trade-off: produces one paragraph of free-text comparison rather than a structured 20-row register; see Section 3 below for the side-by-side.

2 — Chain architecture (initial evaluated baseline)

Five-step obligation-level reasoning chain. Implemented in src/chain.py. Cost: ~50,000 tokens, ~2.5 min per cold query on Groq Llama 70B (free-tier daily limit after 2 queries). Preserved in the codebase as the as-evaluated baseline that supports the report's findings; not the production demo path.

Input
User question
CHN-01 · Decompose
1 LLM call
Split user query into focused sub-questions. Cap at 4 sub-questions (decisions.md §3). Typical: 3 sub-questions per query.
CHN-02 · Retrieve regulation chunks (no LLM)
Per sub-question
For each sub-question, MiniLM-based retrieval, top-5 chunks from corpus_tag == "REG". Result: ~15–20 regulation chunks total.
CHN-03 · Extract atomic obligations
1 LLM call per sub-question
For each sub-question, send retrieved law chunks to LLM with prompt "extract atomic obligations from these chunks". Each call returns ~5 obligation strings. Result: typically 12–20 obligations total.
CHN-04 phase 1 · Silence detection (no LLM)
Per obligation, deterministic threshold check
For each obligation: embed it, compute max cosine similarity against deployer corpora (DEP + DEP_EXTRAS + ICO operational guidance). If max_sim < τ (default 0.35), classify as silent; skip phase 2. Otherwise pass through.
CHN-04 phase 2 · Classify surviving obligations
1 LLM call per sub-question, batched
For surviving obligations in each sub-question, send obligations + retrieved deployer evidence to LLM with prompt "classify each obligation as silent / partial / adequate / contradictory, with citations". Output: per-obligation classification with chunk_ids.
CHN-05 · Synthesise register
1 LLM call (final)
Takes all per-obligation rows from all sub-questions. Calls LLM with prompt "add a gap_characterisation field to each row describing what aspect of the obligation the policy fails to address". Output: structured register rows.
Output
Residual-risk register (~20 rows)
9-field structured rows per obligation: regulatory_provision, regulation_chunk_ids, obligation, match_status, policy_evidence, extras_evidence, guidance_evidence, gap_characterisation, confidence.

Total LLM calls: 8–10 (1 decompose + N extract + N classify + 1 synthesise)  ·  Token cost: ~50K  ·  Cold latency: ~2.5 min on Llama 70B (Groq)

Why this shape was chosen originally: obligation-level matching makes silences structurally visible per the project's distinctive intellectual claim (D-008 in the spec). Threshold-grounded silence detection (Phase 1) was the spec's differentiator: silence is a retrieval property, not an LLM judgement.

Why we moved away from it: empirical evaluation surfaced three architectural failure modes that retrieval improvement does not fix — decomposition drift, miscalibrated silence detection (τ never empirically tuned), unreliable provenance labelling. Plus free-tier Groq daily limits made iterative development impractical. See evaluation-findings.md Stages 1, 5, 7.
